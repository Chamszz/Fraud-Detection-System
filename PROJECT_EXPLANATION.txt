Project Overview
================
This project is a credit card fraud detection system built around a supervised machine learning model and a FastAPI backend. The dataset is stored in `data/creditcard.csv`, and the fraud label is `is_fraud`.

Key components:
- `config.py`: global settings, file paths, and scoring constants
- `utils/features.py`: feature extraction for each transaction
- `export_user_data.py`: report generation from user transactions
- `backend/main.py`: API endpoints, model scoring, threshold decision, and transaction handling
- `backend/retrain.py`: model training and retraining logic
- `frontend/app.py`: Streamlit user interface
- `user_profiles.db`: SQLite profile database used to compute user baseline features

2. Feature Extraction
---------------------
File: `utils/features.py`

This module transforms a transaction into numeric features used by the model.

Relevant code:
```python
# utils/features.py
feat = process(txn)
```

Core logic:
- `cc = txn['cc_num']`
- `amt = txn['amt']`
- compute transaction time:
  - `t_val = hour + minute / 60 + second / 3600`
- load user baseline from `user_profiles.db` using `get_data(cc_num)`
- if baseline exists:
  - `avg_amt = base['avg_amount']`
  - `freq_t = base['frequent_time']`
- if no baseline:
  - use current amount/time as baseline

Derived features returned:
- `a_dev`: absolute amount deviation as fraction of average amount
- `t_dev`: normalized time deviation relative to user’s frequent time
- `a_log`: `log(amt + 1)`
- `t_val`: decimal time value of transaction

This means the model is not using raw text fields directly, but rather derived numeric features representing
how unusual the transaction amount and time are for that card.


3. Model Training and Retraining
--------------------------------
File: `backend/retrain.py`

Training uses a Random Forest classifier.

Relevant code:
```python
m = RandomForestClassifier(
    n_estimators=250,
    max_depth=12,
    min_samples_split=5,
    min_samples_leaf=3,
    random_state=42,
    class_weight='balanced_subsample',
    n_jobs=-1
)
m.fit(X, y)
```

Training flow:
- Read `data/creditcard.csv`
- parse transaction time into `hour`, `minute`, `second`
- load additional feedback data from `transaction_feedback.db`
- append approved feedback transactions with `is_fraud=0`
- extract features for every transaction using `extractFeatures`
- create `X` feature matrix and `y` label array
- fit the Random Forest model
- save model to `backend/models/model.pkl`

Retraining occurs when a pending transaction is manually approved or rejected.
When this happens, `backend/main.py` calls:
- `retrainModel()`
- `reloadModelInBackground()`

So the model is updated over time when user feedback is logged.


4. Thresholds and Risk Scoring
-----------------------------
File: `config.py`

Threshold constants:
- `riskThreshold = 80` — final risk score threshold for approval vs pending
- `riskAmt3x = 95` — minimum risk for transactions 3× above average
- `risk_amount_2x = 90` — minimum risk for transactions 2× above average
- `riskAmtScale = 25` — scale factor for moderate amount increases
- `riskTimeScale = 2` — scale factor for unusual transaction times
- `riskMlMultiplier = 200` — not used directly in current scoring logic
- `riskyAmtRatio = 2.0` — business rule ratio for risky amounts

These values are fixed business-rule parameters, not learned by the Random Forest. They are chosen to define how strictly the system reacts to large deviations from normal spending behavior.

Why these specific values?
- `riskThreshold = 80` means the system treats anything below 80 as acceptable risk and approves it.
- `riskAmt3x = 95` and `risk_amount_2x = 90` force very high risk when a transaction is far above the user’s average amount.
- `riskAmtScale = 25` adds extra risk for amounts above average but below the 2× threshold.
- `riskTimeScale = 2` adds risk when a transaction time is unusual for the cardholder.

These thresholds are applied after the model produces a fraud probability. The model is responsible for scoring how suspicious the transaction is from the learned data; the threshold constants are used to adjust the score with domain-specific fraud rules.

File: `backend/main.py`

Relevant logic:
```python
feat = extractFeatures(txn.dict())
fraud_prob = model.predict_proba([feat])[0][1]
risk, reason = calc_risk(fraud_prob, txn.amt, info)
```

This shows the model output is a probability, not a final pass/fail label.

In `calc_risk(...)`:
```python
def calc_risk(fraud_prob, amt, baseline):
    risk = fraud_prob * 100
    reason = ""
    if baseline:
        ratio = amt / baseline['avg_amount'] if baseline['avg_amount'] > 0 else 1.0
        t_val = baseline.get('frequent_time', 0)
        t_raw = abs((0 - t_val))
        t_diff = min(t_raw, 24 - t_raw)
        if ratio >= 3:
            risk = max(risk, riskAmt3x)
            reason += "Amount 3x+ above average. "
        elif ratio >= 2:
            risk = max(risk, risk_amount_2x)
            reason += "Amount 2x+ above average. "
        elif ratio > 1:
            risk = min(100, risk + int((ratio - 1.0) * riskAmtScale))
        risk = min(100, risk + int(t_diff * riskTimeScale))
        if t_diff > 1.0:
            reason += "Unusual transaction time. "
    return int(risk), reason.strip() if reason else "Flagged by system"
```

How fraud detection works in practice:
- the Random Forest model learns fraud probability from historical transactions and derived features
- this probability is scaled to a 0-100 risk score
- the code then increases that score for strong anomalies: very large amounts or strange times
- this combined risk score is the actual number compared against `riskThreshold`

Exactly where the threshold is used:
```python
if risk < riskThreshold:
    status = "APPROVED"
else:
    status = "PENDING"
```

The model itself does not receive the threshold directly. Instead, the model produces `fraud_prob`, and the threshold is applied in `backend/main.py` after the score is adjusted.

So the fraud transaction decision path is:
1. model predicts fraud probability from features
2. `calc_risk(...)` converts that probability into a risk score
3. business rules adjust risk using `config.py` values
4. final risk is compared to `riskThreshold`
5. transaction is approved or held for review

This means the threshold values are part of the decision engine, not part of the learned model weights.


5. API and Transaction Flow
---------------------------
File: `backend/main.py`

Endpoints:
- `GET /validate/{cc_num}`
  - checks card validity
  - returns card holder profile fields
- `POST /transaction`
  - validates card existence
  - extracts features
  - predicts fraud probability
  - computes risk score
  - decides APPROVED vs PENDING
  - logs transaction to history
- `POST /transaction/{tid}/resolve`
  - approve or reject a pending transaction
  - logs feedback
  - triggers retraining and model reload
- `GET /history/{cc_num}`
  - returns recent transaction history and today’s spending

Example decision flow in `/transaction`:
1. Validate card exists in `user_profiles.db`
2. Build feature vector from transaction data
3. Run `model.predict_proba([feat])`
4. Convert probability into `risk`
5. Apply baseline adjustments and time penalties
6. Compare to `riskThreshold`
7. Return `APPROVED` or `PENDING`

If `PENDING`, the transaction is stored in `pending` memory until resolved.

The database is loaded from `user_profiles.db` in the repository root.
To build it, an initialization script like `init_db.py` can be used. This step is required before using the API if the DB is missing.

Displayed outputs:

11. How model improvement happens over time
------------------------------------------
- User feedback on pending transactions is stored in `transaction_feedback.db`
- The next retraining run includes these approved cases as new non-fraud examples
- The saved model file `backend/models/model.pkl` is replaced with the updated Random Forest
- `backend/main.py` reloads the model in the background so future requests use the newest version

Because of this, the system is not static: it adapts based on approval feedback.

9. How the model improves over time
-----------------------------------
Retraining is triggered when pending transactions are resolved in `/transaction/{tid}/resolve`.
The backend logs feedback to `transaction_feedback.db` and then runs `retrainModel()`.
The retraining procedure:
- reads original dataset
- reads feedback-approved transactions
- adds approved transactions as non-fraud examples (`is_fraud=0`)
- retrains the Random Forest on the expanded data
- saves the updated model
- reloads the model in the running API

This means the system can adapt when users approve suspicious transactions.


10. Summary
-----------
The system works as follows:
- raw transaction data and user profiles are stored in files and databases
- features are extracted from amount and time relative to user history
- Random Forest predicts fraud probability from those features
- custom rule-based risk scoring adjusts probability using business thresholds
- `riskThreshold` controls whether a transaction is approved or flagged
- pending transactions can be reviewed and used to retrain the model
- a Streamlit frontend provides UI for transaction entry, mobile approval, and history

Each part of the code has a precise role:
- `config.py`: thresholds and paths
- `utils/features.py`: feature generation and baseline lookup
- `backend/retrain.py`: model training and retraining
- `backend/main.py`: scoring, thresholding, and API logic
- `frontend/app.py`: user-facing interface
- `export_user_data.py`: analytics report generation

This document is meant to explain project flow clearly, with file paths and code behavior for the exact fraud scoring and decision process.
